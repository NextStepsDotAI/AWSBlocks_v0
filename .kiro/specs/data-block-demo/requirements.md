# Requirements Document

## Introduction

The Data Block Demo is the first scenario in the AWS Blocks showcase project. It demonstrates data persistence capabilities by implementing a simple todo-item CRUD microservice twice: once using the Data Block (`blocks/data/`) and once using a Reference Implementation that uses raw AWS CDK and a Lambda with AWS SDK v3 (`reference-implementation/data/`).

The demo's primary goal is to make the productivity and simplicity difference between the two approaches explicit and measurable. The target audience is a team of 300 developers who build serverless microservices. Both implementations are functionally equivalent: they expose the same CRUD operations on the same todo-item data model and produce identical API behaviour from the consumer's perspective.

Important deployment constraint:
- Both the Blocks-based implementation and the Reference Implementation SHALL be deployed using AWS CDK. The Blocks_Service must expose or include a CDK stack (a "CDK wrapper") that instantiates the Data_Block's deployment constructs so that both implementations are deployable via the same `cdk synth` / `cdk deploy` workflow and the same enterprise CDK toolchain.

## Glossary

- **Data_Block**: The `@aws-blocks/data` npm package that provides a typed application API for data persistence, backed locally by Postgres and by DynamoDB on AWS deployment.
- **Reference_Stack**: The raw AWS CDK stack defined in `reference-implementation/data/` that manually provisions a DynamoDB table and wires a Lambda handler using the AWS SDK v3 DynamoDB client — representing the team's manual approach.
- **Blocks_Service**: The AWS Blocks-based todo-item service implemented in `blocks/data/` using the Data_Block.
- **Blocks_CDK_Wrapper**: A CDK stack (source kept in the repo under an agreed path, e.g., `infra/blocks/data-stack.ts`) which provisions any AWS resources required by the Blocks_Service by instantiating Data_Block-provided CDK constructs or using the Data_Block's documented CDK integration points. This wrapper ensures Blocks-based resources are managed through the enterprise CDK workflow.
- **Reference_Service**: The Lambda-based todo-item service implemented in `reference-implementation/data/` using the Reference_Stack.
- **TodoItem**: The shared data model representing a todo item, with fields: `id` (string, UUID v4), `title` (string), `completed` (boolean), `createdAt` (ISO-8601 UTC timestamp string).
- **CRUD_Operations**: The four data operations exposed by both services — Create (POST /todos), Read (GET /todos/{id}), List (GET /todos), and Update (PATCH /todos/{id}). Update uses the HTTP PATCH verb and partial update semantics. (Delete is intentionally excluded in this demo.)
- **Local_Dev_Mode**: Running the demo with `npm run dev` against local infrastructure (Postgres for the Blocks_Service) with no AWS account, credentials, or CDK bootstrap required.
- **Side_By_Side_Summary**: The `README.md` comparison table in `reference-implementation/data/` that presents the quantitative productivity metrics between the two approaches.
- **Metrics_Report**: The machine-readable JSON or Markdown artifact (generated by a script in `scripts/`) containing the counted productivity metrics: lines of code, file count, manually configured AWS resources, and tool dependencies.

## API Contract (explicit)

All endpoints return JSON and use the error response shape defined below.

- POST /todos
  - Purpose: Create a new TodoItem
  - Request body: { "title": string }
  - Validation: title must be non-empty (non-whitespace) and ≤ 200 characters
  - Success: 201 Created with body: TodoItem
  - Error: 400 ValidationError

- GET /todos/{id}
  - Purpose: Retrieve a single TodoItem by id
  - Success: 200 OK with body: TodoItem
  - Error: 404 NotFound

- GET /todos
  - Purpose: List all TodoItem records
  - Success: 200 OK with body: TodoItem[] ordered by `createdAt` descending

- PATCH /todos/{id}
  - Purpose: Partially update an existing TodoItem
  - Request body: { "title"?: string, "completed"?: boolean }
  - Validation: At least one of `title` or `completed` must be present. If `title` is present it must be non-empty and ≤ 200 characters.
  - Success: 200 OK with body: TodoItem
  - Error: 400 ValidationError or 404 NotFound

Notes:
- Delete is intentionally omitted from the demo (CRUD here = Create, Read, List, Update).
- All request and response JSON use UTF-8 and standard content-type headers.

Error response shape (for 4xx and 5xx):
```json
{
  "error": {
    "code": "ValidationError" | "NotFound" | "InternalError",
    "message": "human-readable description"
  }
}
```

## Requirements

### Requirement 1: Blocks-Based Todo-Item Service

**User Story:** As a developer evaluating AWS Blocks, I want to see a fully working CRUD microservice implemented using the Data Block, so that I can understand the Data Block's API surface and its developer ergonomics.

#### Acceptance Criteria

1. THE Blocks_Service SHALL implement Create, Read (by id), List, and Update operations for TodoItem entities following the API contract above.
2. WHEN a Create operation is invoked with a valid non-empty title string of ≤ 200 characters, THE Blocks_Service SHALL persist a new TodoItem with:
   - an id generated as a UUID v4 string,
   - the provided title,
   - completed defaulting to false,
   - createdAt set to the current UTC time as an ISO-8601 string.
3. WHEN a Read operation is invoked with an existing TodoItem id, THE Blocks_Service SHALL return the corresponding TodoItem.
4. IF a Read operation is invoked with an id that does not exist, THEN THE Blocks_Service SHALL return a not-found error response with HTTP status 404 and the error shape above.
5. WHEN a List operation is invoked, THE Blocks_Service SHALL return all persisted TodoItem records in descending order of `createdAt`.
6. WHEN an Update operation is invoked with an existing TodoItem id and a valid partial update payload, THE Blocks_Service SHALL persist only the provided fields (limited to `title` and `completed`) and return the updated TodoItem.
7. IF an Update operation is invoked with an id that does not exist, THEN THE Blocks_Service SHALL return a not-found error response with HTTP status 404.
8. THE Blocks_Service SHALL use only Data_Block typed APIs for all persistence operations — no raw DynamoDB SDK calls and no raw SQL queries SHALL appear in `blocks/data/` source files.
9. THE Blocks_Service SHALL expose all CRUD operations through the Data_Block's type-safe API surface so that TypeScript types for request and response shapes are derived from the Block definition.
10. IF a Create operation is invoked with an empty string or whitespace-only title, THEN THE Blocks_Service SHALL return a validation error with HTTP status 400.
11. IF an Update operation is invoked with a payload where all fields are undefined or absent, THEN THE Blocks_Service SHALL return a validation error with HTTP status 400.
12. THE Blocks_Service SHALL include a CDK wrapper stack (the Blocks_CDK_Wrapper) stored in the repository so it can be synthesized and deployed via the enterprise CDK toolchain. This wrapper SHALL either instantiate Data_Block-provided CDK constructs or use the documented Data_Block integration so that all AWS resources required by the Blocks_Service are created through CDK.
12.a THE Blocks_CDK_Wrapper SHALL set CloudWatch Log Group retention to logs.RetentionDays.ONE_DAY for all Lambda log groups the wrapper provisions or for Block-provisioned Lambda log groups where the Block exposes the log-group construct.
12.b THE Blocks_CDK_Wrapper SHALL set RemovalPolicy.DESTROY on stateful non-production resources it creates (DynamoDB tables, S3 buckets, etc.). Any production exceptions must be explicitly documented in the stack file with a justification comment.

### Requirement 2: Reference Implementation (raw CDK / Lambda)

**User Story:** As a developer evaluating AWS Blocks, I want to see the same todo-item CRUD microservice implemented using our team's current raw CDK and AWS SDK v3 approach, so that I have a direct before/after comparison.

#### Acceptance Criteria

1. THE Reference_Service SHALL implement the same Create, Read (by id), List, and Update operations for TodoItem entities as the Blocks_Service, producing identical API behaviour from a consumer perspective per the API contract.
2. THE Reference_Stack SHALL define the DynamoDB table using an explicit `aws-cdk-lib/aws-dynamodb` `Table` construct with `PAY_PER_REQUEST` billing mode and a string partition key named `PK`.
3. THE Reference_Stack SHALL define the Lambda handler using an explicit `aws-cdk-lib/aws-lambda` `Function` construct with Node.js 24.x runtime, `memorySize` of 512 MB, and `timeout` of 29 seconds.
3.a THE Reference_Stack SHALL set reservedConcurrentExecutions on every Lambda function according to POC defaults: dev=5, staging=20, prod=100. This value MUST be configurable via stack props and documented in the stack file.
3.b THE Reference_Stack SHALL configure API stage-level throttling for any HTTP API with POC defaults: dev rate=10 req/s burst=20, staging rate=50 burst=100, prod rate=200 burst=400 (use the same throttle values defined in infra-patterns.md).
3.c THE Reference_Stack SHALL ensure CloudWatch Log Groups created for Lambda functions have a retention policy of logs.RetentionDays.ONE_DAY and an appropriate removalPolicy so they are destroyed in non-prod.
4. THE Reference_Service Lambda handler SHALL use the `@aws-sdk/client-dynamodb` or `@aws-sdk/lib-dynamodb` AWS SDK v3 package for all DynamoDB operations — no other ORM or persistence abstraction permitted.
5. THE Reference_Stack SHALL apply all mandatory resource tags (`Project`, `Environment`, `Service`, `Owner`, `ManagedBy`, `CostCenter`) at the stack level using `cdk.Tags.of(this)`.
6. THE Reference_Stack SHALL use `RemovalPolicy.DESTROY` for the DynamoDB table so the demo stack can be cleanly torn down.
7. THE Reference_Service SHALL use `@aws-lambda-powertools/logger` for structured JSON logging, including the `correlationId` in every log entry (where correlationId is read from the `x-correlation-id` header when present).
8. IF the Reference_Service Lambda handler receives a request with a missing or malformed required field, THEN it SHALL return HTTP 400 with the ValidationError error shape.
9. IF the Reference_Service Lambda handler encounters an unhandled runtime error, THEN it SHALL log the error and return an HTTP 500 response with the InternalError shape.
10. THE TodoItem entity, as used in Reference_Service responses, SHALL conform to the agreed shared shape `{ id: string, title: string, completed: boolean, createdAt: string }`.
11. THE Reference_Stack SHALL be synthesized and deployed via the same enterprise CDK workflow and commands used for the Blocks_CDK_Wrapper (e.g., `npm run cdk:synth` and `npm run cdk:deploy` should work for both stacks).

### Requirement 3: Shared TodoItem Data Model

**User Story:** As a developer reading the demo code, I want both implementations to use the same TypeScript type definition for the todo item, so that I can confirm the two services are genuinely equivalent.

#### Acceptance Criteria

1. THE Demo SHALL define the TodoItem TypeScript interface once in `shared/models/todo-item.ts` and import it in both `blocks/data/` and `reference-implementation/data/` — neither implementation may re-declare the shape.
2. THE TodoItem interface SHALL include the fields `id: string`, `title: string`, `completed: boolean`, and `createdAt: string` where `createdAt` is an ISO 8601 UTC date-time string.
3. THE Demo SHALL define a `CreateTodoRequest` type containing `title: string` and an `UpdateTodoRequest` type containing `title?: string` and `completed?: boolean` in the same `shared/models/todo-item.ts` file.

### Requirement 4: Local Development Without AWS Account

**User Story:** As a developer evaluating AWS Blocks, I want to run the Blocks-based demo locally with a single command and no AWS credentials.

#### Acceptance Criteria

1. WHEN a developer runs `npm run dev` from the project root, THE Blocks_Service SHALL start, serve all CRUD operations against a local Postgres instance managed by the Data_Block, and emit a readiness log entry **within 30 seconds**. If the local Postgres instance is temporarily unavailable, the service may retry with an exponential backoff but must fail fast (exit non-zero) if the database is still unreachable after 30 seconds; this retry strategy must be documented in `blocks/data/README.md`.
2. THE Blocks_Service source code SHALL NOT contain any environment-specific branches that rely on `process.env.NODE_ENV`, `process.env.ENVIRONMENT`, or similar to switch persistence mechanisms — Blocks must handle environment differences.
3. THE Demo repository SHALL include a `README.md` in `blocks/data/` that provides the exact `npm run dev` command and lists local prerequisites (e.g., Docker for Postgres) with minimum versions and startup instructions.

---

### Requirement 5: AWS Deployment Without Code Changes (Both Implementations via CDK)

**User Story:** As a developer evaluating AWS Blocks, I want to deploy both the Blocks-based service and the Reference Implementation to AWS using the same CDK-based enterprise workflow so that deployment is consistent and auditable.

#### Acceptance Criteria

1. WHEN either service is deployed to AWS (via `npm run cdk:deploy`), THE deployment SHALL be performed via CDK — both the Blocks_CDK_Wrapper and the Reference_Stack must synthesize and deploy through the enterprise CDK workflow without manual console steps.
2. FOR the Blocks_Service, the Blocks_CDK_Wrapper SHALL provision DynamoDB and all other AWS resources required by the service by instantiating Data_Block's CDK constructs or documented integration points — no manually authored CDK DynamoDB or Lambda constructs outside the wrapper are required to make the Blocks_Service deployable.
3. BOTH the Blocks_CDK_Wrapper and the Reference_Stack SHALL apply mandatory resource tags and POC guardrails (log retention, reserved concurrency, throttling defaults) at stack creation time so policy and cost controls are enforceable via CDK.
4. WHEN a deployed Blocks_Service receives a Create, Read, List, or Update request, THE Blocks_Service SHALL produce the same response structure as it does in Local_Dev_Mode (see API contract).
5. IF `npm run cdk:deploy` fails due to missing AWS credentials or an un-bootstrapped CDK environment, THEN the tooling SHALL emit an actionable error message identifying the specific missing prerequisite.

### Requirement 6: Side-By-Side Productivity Comparison

**User Story:** As a developer evaluating AWS Blocks, I want to see a concrete, quantified comparison of both implementations, so that I can form a data-driven opinion.

#### Acceptance Criteria

1. THE Demo SHALL include a `README.md` in `reference-implementation/data/` containing a Side_By_Side_Summary table that presents the following metrics for both approaches: total lines of application code (excluding tests and config), file count, number of manually authored infrastructure constructs, and required developer tools for local dev.
2. THE Side_By_Side_Summary SHALL include a row explicitly counting the DynamoDB SDK call lines (put, get, query, update) present in the Reference_Service and the equivalent line count (0 or minimal) in the Blocks_Service.
3. THE Side_By_Side_Summary SHALL include a row for local dev complexity listing required tools for each approach.
4. THE Demo SHALL include a `scripts/count-metrics.ts` script that counts lines of code and files for both `blocks/data/` and `reference-implementation/data/` applying the specified exclusions.
5. THE Script SHALL exclude test files (`*.test.ts`, `*.spec.ts`) and configuration files (`*.json`, `*.yaml`, `*.yml`) from all line counts.
6. WHEN the `scripts/count-metrics.ts` script is executed against the committed codebase, the metric values it prints SHALL match the Side_By_Side_Summary table; the process producing the table must be deterministic and documented.

### Requirement 7: Test Coverage for Both Implementations

**User Story:** As a developer reviewing the demo, I want both the Blocks-based and reference services to have tests, so that I can confirm implementations are correct and the comparison is fair.

#### Acceptance Criteria

1. THE Blocks_Service SHALL have unit tests in `test/data/blocks/` covering Create, Read, List, and Update operations with representative cases (happy path, validation failure, not-found, and an error path) — document the exact test scenarios in the test README.
2. THE Reference_Service Lambda handler SHALL have unit tests in `test/data/reference/` with representative scenarios per CRUD operation (happy path and validation failures).
3. THE Reference_Stack and the Blocks_CDK_Wrapper SHALL each have a CDK assertion test in `test/data/reference/` or `test/data/blocks/` that verifies a DynamoDB table and a Lambda function with Node.js 24.x runtime are synthesised (as applicable).
4. THE Demo test suite SHALL NOT make real AWS SDK calls or real DynamoDB calls — all AWS interactions SHALL be mocked at the SDK or Block boundary.
5. WHEN a valid TodoItem is serialized to JSON via `JSON.stringify` and deserialized via `JSON.parse`, THE result SHALL deep-equal the original TodoItem object.
6. THE Demo test suite SHALL achieve at least 80% line coverage across `blocks/data/` and `reference-implementation/data/` source files, measured by `npm run test:coverage`.

---

### Requirement 8: Demo Narrative and Inline Documentation

**User Story:** As a developer attending an AWS Blocks showcase session, I want the demo code and README files to explain the "so what" of each implementation choice.

#### Acceptance Criteria

1. THE `blocks/data/` directory SHALL contain a `README.md` with sections titled "What the Data Block provides", "AWS services replaced", "Running locally", and "Deploying to AWS" that includes CDK deployment instructions for the Blocks_CDK_Wrapper.
2. THE `reference-implementation/data/` directory SHALL contain a `README.md` with sections titled "Manual steps required", "Problems with this approach", and "See the comparison", and SHALL include CDK deployment instructions.
3. THE Reference_Stack source file SHALL include inline comments labelled `// CURRENT APPROACH:` before the Table construct, Function construct, IAM grant call, and resource tagging block.
4. THE Reference_Service Lambda handler source file SHALL include inline comments labelled `// RAW SDK:` before the putItem, getItem, query, and updateItem SDK calls.
5. THE `blocks/data/` source files SHALL include inline comments labelled `// BLOCKS APPROACH:` at the point where the Data_Block is instantiated and at each CRUD operation call.
6. THE Blocks_CDK_Wrapper source file SHALL document how tags and POC guardrails are applied to Block-provisioned resources (and how to override defaults via documented configuration or an escape-hatch).
