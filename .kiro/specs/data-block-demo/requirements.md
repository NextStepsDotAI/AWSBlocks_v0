# Requirements Document

## Introduction

The Data Block Demo is the first scenario in the AWS Blocks showcase project. It demonstrates data persistence capabilities by implementing a simple todo-item CRUD microservice twice: once using the AWS Blocks Data persistence Block (`blocks/data/`) and once using the team's current approach of raw AWS CDK DynamoDB constructs with AWS SDK v3 client calls from Lambda handlers (`comparison/data/`).

The demo's primary goal is to make the productivity and simplicity difference between the two approaches explicit and measurable. The target audience is a team of 300 developers who build serverless microservices with raw CDK, Lambda, API Gateway, and DynamoDB. They must be able to see — in concrete, counted terms — how many lines of CDK and SDK code the Data Block eliminates and how much simpler local development becomes.

Both implementations are functionally equivalent: they expose the same CRUD operations on the same todo-item data model and produce identical API behaviour from the consumer's perspective.

---

## Glossary

- **Data_Block**: The `@aws-blocks/data` npm package that provides a typed application API for data persistence, backed locally by Postgres and by DynamoDB on AWS deployment.
- **Comparison_Stack**: The raw AWS CDK stack defined in `comparison/data/` that manually provisions a DynamoDB table and wires a Lambda handler using the AWS SDK v3 DynamoDB client — representing the team's current approach.
- **Blocks_Service**: The AWS Blocks-based todo-item service implemented in `blocks/data/` using the Data_Block.
- **Comparison_Service**: The Lambda-based todo-item service implemented in `comparison/data/` using the Comparison_Stack.
- **TodoItem**: The shared data model representing a todo item, with fields: `id` (string, UUID), `title` (string), `completed` (boolean), `createdAt` (ISO-8601 timestamp string).
- **CRUD_Operations**: The four data operations exposed by both services — Create, Read (single item), List (all items), and Update.
- **Local_Dev_Mode**: Running the demo with `npm run dev` against local infrastructure (Postgres for the Blocks_Service) with no AWS account, credentials, or CDK bootstrap required.
- **Side_By_Side_Summary**: The `README.md` comparison table in `comparison/data/` that presents the quantitative productivity metrics between the two approaches.
- **Metrics_Report**: The machine-readable JSON or Markdown artifact (generated by a script in `scripts/`) containing the counted productivity metrics: lines of code, file count, manually configured AWS services, and local dev steps.

## Requirements

### Requirement 1: Blocks-Based Todo-Item Service

**User Story:** As a developer evaluating AWS Blocks, I want to see a fully working CRUD microservice implemented using the Data Block, so that I can understand the Data Block's API surface and the minimal code required to persist and retrieve data.

#### Acceptance Criteria

1. THE Blocks_Service SHALL implement Create, Read (by id), List, and Update operations for TodoItem entities.
2. WHEN a Create operation is explicitly invoked with a valid non-empty title string of ≤ 200 characters, THE Blocks_Service SHALL persist a new TodoItem with a generated UUID id, the provided title, `completed` set to `false`, and `createdAt` set to the current UTC timestamp as an ISO 8601 date-time string (e.g. `2024-01-15T10:30:00.000Z`).
3. WHEN a Read operation is invoked with an existing TodoItem id, THE Blocks_Service SHALL return the corresponding TodoItem.
4. IF a Read operation is invoked with an id that does not exist, THEN THE Blocks_Service SHALL return a not-found error response with HTTP status 404.
5. WHEN a List operation is invoked, THE Blocks_Service SHALL return all persisted TodoItem records in descending order of `createdAt`.
6. WHEN an Update operation is invoked with an existing TodoItem id and a valid partial update payload, THE Blocks_Service SHALL persist the updated fields — limited to `title` and `completed` only — and return the complete updated TodoItem.
7. IF an Update operation is invoked with an id that does not exist, THEN THE Blocks_Service SHALL return a not-found error response with HTTP status 404.
8. THE Blocks_Service SHALL use only Data_Block typed APIs for all persistence operations — no raw DynamoDB SDK calls and no raw SQL queries SHALL appear in `blocks/data/` source files.
9. THE Blocks_Service SHALL expose all CRUD operations through the Data_Block's type-safe API surface so that TypeScript types for request and response shapes are derived from the Block definition rather than written manually.
10. IF a Create operation is invoked with an empty string or whitespace-only title, THEN THE Blocks_Service SHALL return a validation error with HTTP status 400.
11. IF an Update operation is invoked with a payload where all fields are undefined or absent, THEN THE Blocks_Service SHALL return a validation error with HTTP status 400.

### Requirement 2: Comparison Lambda-Based Todo-Item Service

**User Story:** As a developer evaluating AWS Blocks, I want to see the same todo-item CRUD microservice implemented using our team's current raw CDK and AWS SDK v3 approach, so that I have a direct like-for-like baseline to compare against.

#### Acceptance Criteria

1. THE Comparison_Service SHALL implement the same Create, Read (by id), List, and Update operations for TodoItem entities as the Blocks_Service, producing identical API behaviour from a consumer perspective, with the following HTTP contract: POST /todos returns 201, GET /todos/:id returns 200, GET /todos returns 200, and PUT /todos/:id returns 200.
2. THE Comparison_Stack SHALL define the DynamoDB table using an explicit `aws-cdk-lib/aws-dynamodb` `Table` construct with `PAY_PER_REQUEST` billing mode and a string partition key named `PK`.
3. THE Comparison_Stack SHALL define the Lambda handler using an explicit `aws-cdk-lib/aws-lambda` `Function` construct with Node.js 24.x runtime, `memorySize` of 512 MB, `timeout` of 29 seconds, and `reservedConcurrentExecutions` set to 5 for dev.
4. THE Comparison_Service Lambda handler SHALL use the `@aws-sdk/client-dynamodb` or `@aws-sdk/lib-dynamodb` AWS SDK v3 package for all DynamoDB operations — no other ORM or persistence abstraction SHALL be used.
5. THE Comparison_Stack SHALL apply all mandatory resource tags (`Project`, `Environment`, `Service`, `Owner`, `ManagedBy`, `CostCenter`) at the stack level using `cdk.Tags.of(this)`.
6. THE Comparison_Stack SHALL use `RemovalPolicy.DESTROY` for the DynamoDB table so the demo stack can be cleanly torn down.
7. THE Comparison_Service SHALL use `@aws-lambda-powertools/logger` for structured JSON logging, including the `correlationId` in every log entry, where `correlationId` is read from the `x-correlation-id` request header and, if absent, a UUID is generated and used.
8. IF the Comparison_Service Lambda handler receives a request with a missing or malformed required field — where Create requires `title` and Update requires at least one of `title` or `completed` — THEN THE Comparison_Service SHALL return an HTTP 400 response with a structured error body.
9. IF the Comparison_Service Lambda handler encounters an unhandled runtime error, THEN THE Comparison_Service SHALL log the error and return an HTTP 500 response with a structured error body of the form `{ error: string, correlationId: string }` — no unhandled rejection SHALL propagate.
10. THE TodoItem entity, as used in Comparison_Service responses, SHALL conform to the shape `{ id: string, title: string, completed: boolean, createdAt: string }` as defined in the shared TodoItem interface.

### Requirement 3: Shared TodoItem Data Model

**User Story:** As a developer reading the demo code, I want both implementations to use the same TypeScript type definition for the todo item, so that I can confirm the two services are genuinely equivalent and any type differences are artefacts of the implementation approach rather than the model.

#### Acceptance Criteria

1. THE Demo SHALL define the TodoItem TypeScript interface once in `shared/models/todo-item.ts` and import it in both `blocks/data/` and `comparison/data/` — neither `blocks/data/` nor `comparison/data/` source files SHALL declare a local TodoItem interface.
2. THE TodoItem interface SHALL include the fields `id: string`, `title: string`, `completed: boolean`, and `createdAt: string`, where `createdAt` is an ISO 8601 date-time string.
3. THE Demo SHALL define a `CreateTodoRequest` type containing `title: string` and a `UpdateTodoRequest` type containing `title?: string` and `completed?: boolean` in the same `shared/models/todo-item.ts` file.

### Requirement 4: Local Development Without AWS Account

**User Story:** As a developer evaluating AWS Blocks, I want to run the Blocks-based demo locally with a single command and no AWS credentials, so that I can experience the local dev story first-hand and contrast it with the comparison approach.

#### Acceptance Criteria

1. WHEN a developer runs `npm run dev` from the project root, THE Blocks_Service SHALL start, serve all CRUD operations against a local Postgres instance managed by the Data_Block, and emit a readiness signal to stdout within 30 seconds — no AWS account, credentials, region configuration, or CDK bootstrap SHALL be required.
2. THE Blocks_Service source code SHALL NOT contain any environment-specific branches — defined as conditional expressions whose condition references `process.env.NODE_ENV`, `process.env.ENVIRONMENT`, or any variable whose value identifies the runtime environment — that switch between local and AWS behaviour; the Data_Block SHALL handle environment switching internally.
3. THE Demo repository SHALL include a `README.md` in `blocks/data/` that provides the exact `npm run dev` command, lists the local prerequisites (e.g. Docker for Postgres) with the minimum version of each prerequisite, and confirms no AWS account is needed.
4. IF the local Postgres instance is unavailable when `npm run dev` is executed, THEN THE Blocks_Service SHALL exit with a non-zero exit code within 10 seconds and print an error to stderr naming Postgres as the missing dependency.

### Requirement 5: AWS Deployment Without Code Changes

**User Story:** As a developer evaluating AWS Blocks, I want to deploy the Blocks-based service to AWS using the same source code that runs locally, so that I can confirm the "write once, run anywhere" claim of AWS Blocks.

#### Acceptance Criteria

1. WHEN the Blocks_Service is deployed to AWS (via `npm run cdk:deploy`), THE Blocks_Service SHALL use DynamoDB as its persistence backend — no code change, environment variable override, or configuration file edit SHALL be required between local and deployed execution.
2. THE Blocks_Service deployment SHALL not require any manually authored CDK DynamoDB construct, Lambda construct, or API Gateway construct — the Data_Block SHALL provision all required AWS resources automatically.
3. WHEN the deployed Blocks_Service receives a Create, Read, List, or Update request, THE Blocks_Service SHALL produce the same response structure as it does in Local_Dev_Mode, where "same response structure" means identical field names, field types, and HTTP status codes as defined in the TodoItem interface and the HTTP contracts specified in Requirement 2 AC1.
4. IF `npm run cdk:deploy` fails due to missing AWS credentials or an un-bootstrapped CDK environment, THEN THE tooling SHALL emit an actionable error message identifying the specific missing prerequisite.

### Requirement 6: Side-By-Side Productivity Comparison

**User Story:** As a developer evaluating AWS Blocks, I want to see a concrete, quantified comparison of both implementations, so that I can form a data-driven opinion on the productivity and simplicity difference.

#### Acceptance Criteria

1. THE Demo SHALL include a `README.md` in `comparison/data/` containing a Side_By_Side_Summary table that presents the following metrics for both approaches: total lines of application code (excluding blank lines and comments), total lines of infrastructure code, number of source files, number of AWS services manually configured in CDK, and local development steps required.
2. THE Side_By_Side_Summary SHALL include a row explicitly counting the DynamoDB SDK call lines (put, get, query, update) present in the Comparison_Service and the equivalent line count in the Blocks_Service — IF the Blocks_Service contains zero raw DynamoDB SDK call lines, THEN the Side_By_Side_Summary SHALL record "0" for the Blocks_Service SDK call count.
3. THE Side_By_Side_Summary SHALL include a row for local dev complexity: the Comparison_Service row SHALL list the tools required (SAM CLI, AWS credentials configuration, DynamoDB Local), and the Blocks_Service row SHALL state "none — Data Block manages local Postgres".
4. THE Demo SHALL include a `scripts/count-metrics.ts` script that counts lines of code and files for both `blocks/data/` and `comparison/data/` and prints the results using the same row labels and metric names as the Side_By_Side_Summary table, so that the metrics can be independently verified and are not manually maintained.
5. THE Script SHALL exclude test files (`*.test.ts`, `*.spec.ts`) and configuration files (`*.json`, `*.yaml`, `*.yml`) from all line counts — this exclusion SHALL apply both when the script is executed and when metrics are manually verified against the Side_By_Side_Summary table.
6. WHEN the `scripts/count-metrics.ts` script is executed against the committed codebase, the metric values it prints SHALL exactly match the values in the Side_By_Side_Summary table with zero tolerance for discrepancy.

### Requirement 7: Test Coverage for Both Implementations

**User Story:** As a developer reviewing the demo, I want both the Blocks-based and comparison services to have tests, so that I can confirm the implementations are correct and the comparison is fair.

#### Acceptance Criteria

1. THE Blocks_Service SHALL have unit tests in `test/data/blocks/` covering the Create, Read, List, and Update operations — with a minimum of 4 test cases per operation including happy path, not-found (for Read and Update), and validation failure (for Create and Update) — mocked at the Data_Block boundary.
2. THE Comparison_Service Lambda handler SHALL have unit tests in `test/data/comparison/` with at least: one happy-path test per CRUD operation, one validation-failure test for missing required fields, and one error-handling test for SDK-level failures, where "SDK-level failures" means the `@aws-sdk/lib-dynamodb` `send()` method rejects with an Error.
3. THE Comparison_Stack SHALL have a CDK assertion test in `test/data/comparison/` that verifies: a DynamoDB table is synthesised, a Lambda function with Node.js 24.x runtime is synthesised, and the Lambda function's execution role has DynamoDB read/write permissions — defined as an IAM policy with `dynamodb:PutItem`, `dynamodb:GetItem`, `dynamodb:Query`, and `dynamodb:UpdateItem` actions scoped to the table ARN.
4. THE Demo test suite SHALL NOT make real AWS SDK calls or real DynamoDB calls — all AWS interactions SHALL be mocked at the SDK or Block boundary.
5. WHEN a valid TodoItem is serialised to JSON via `JSON.stringify` and deserialised via `JSON.parse`, THE result SHALL deep-equal the original TodoItem object.
6. THE Demo test suite SHALL achieve at least 80% line coverage across `blocks/data/` and `comparison/data/` source files, measured by `npm run test:coverage`.

---

### Requirement 8: Demo Narrative and Inline Documentation

**User Story:** As a developer attending an AWS Blocks showcase session, I want the demo code and README files to explain the "so what" of each implementation choice, so that I can follow the narrative without a presenter walking me through every line.

#### Acceptance Criteria

1. THE `blocks/data/` directory SHALL contain a `README.md` with sections titled "What the Data Block provides", "AWS services replaced", "Running locally", and "Deploying to AWS".
2. THE `comparison/data/` directory SHALL contain a `README.md` with sections titled "Manual steps required", "Problems with this approach", and "See the comparison".
3. THE Comparison_Stack source file SHALL include inline comments labelled `// CURRENT APPROACH:` before the Table construct, Function construct, IAM grant call, and resource tagging block.
4. THE Comparison_Service Lambda handler source file SHALL include inline comments labelled `// RAW SDK:` before the putItem, getItem, query, and updateItem SDK calls.
5. THE `blocks/data/` source files SHALL include inline comments labelled `// BLOCKS APPROACH:` at the point where the Data_Block is instantiated and at each CRUD operation call, where each such comment SHALL name at least one specific CDK construct or SDK call that is absent because the Block handles it.
